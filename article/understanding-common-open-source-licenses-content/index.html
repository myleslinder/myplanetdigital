Understanding Common Open Source Licenses

  • technology

You've created a valuable snippet of code and are interested in sharing it with the development community. Where do you begin?

Open source is a philosophy that encourages the free distribution of a software's source code. It allows for collaboration amongst developers, who can then offer the finished product to the community at large. Before beginning an open source initiative, there are a few important questions to consider: Where do you start? Are there legal implications? After open sourcing your code, how does one ensure that it remains free afterwards?

How to Open Source

The process of open sourcing is simple. Start by uploading the project to an online repository. Github is my preferred choice since it allows free online hosting of code and strong support for collaboration between multiple developers (what every open source project desires). Next, add a license file to the repository created and update a README file for referral. Drupal, for example, adds a GPL license to all its contributed projects with a README file mentioning something like: “for legal information see LICENSE.txt in the same directory as this document.” That is it. All open source licenses come with a standard “no warranty” clause so there is no troubling legal issues either.

Choosing a License

Open source licenses are classified as either permissive or copyleft. A copyleft license, such as GPL is viral. It states that all modifications & derivatives of the project or code must also be free & licensed under GPL (or a similar copyleft license).

MIT and BSD are examples of permissive licenses. Here, code is free to use but has no copyleft. Anyone can incorporate the code into their project without obtaining a license that ensures that further code continues to support free projects. These licenses are suitable for commercial use – anyone can sell derivatives of the original code, provided they adhere to the laws enforced by the original license. The difference is in the choice of distribution – a copyleft license requires the release of the source along with the product, whereas a GPL does not. I prefer using GPL for all my projects exactly for this reason because it allows the code to be incorporated into larger projects, with the parent source still available to the public.

Combining Licenses

Incorporating code with existing licenses depends on compatibility. A project under GPL can include MIT code. The combined work will be GPL with the original bit of MIT code, still licensed under MIT. In such cases all licenses must be included. Commercial or not, a project must include copyright notice indicating which MIT libraries it uses.

The Power of Open Source

Open source projects are incredibly valuble not only to developers, but to users. In giving others access to the code, we're encouraging innovation that will contribute to the evolution & betterment of the digital space for everyone.
